# 文档维护基线 v1.0

## 一、目标

统一项目设计文档的术语、结构、验收口径与可执行性，避免后续迭代出现规范漂移。

## 二、统一规范

- 项目命名统一使用“纪光元生”。
- 七层组织架构统一使用 `L0-L6`（`L0` 为老板层）。
- 自上而下层级能力与工具强度以 `LAYER_ABILITY_CONFIG_v1.0.md` 为准；**主脑（L1）**须优先保证工具编排、记忆治理、调度与合规总闸（见该文档「一点五」及 CEO-09～CEO-14）。
- API 规范统一为：基础URL包含 `/api/v1`，端点表默认写相对路径。
- 项目编号统一为 `JYIS-YYYY-XXX`（与 `PROJECT_PLAN_TEMPLATE_v1.0.md`、`PROJECT_MANAGEMENT_MODULE_v1.0.md` 一致）；示例与对话占位符禁止使用 `PRJ-` 前缀。
- 覆盖率口径统一为：全局 `>=80%`，核心模块按 `quality_standards.md` 更高门槛执行。
- MK-01 性能验收统一为：`P95 <= 180秒，单次 <= 300秒`。
- 能力ID以 `AGENT_ABILITY_SPEC_v1.0.md` 为唯一基线，禁止引用未定义ID。
- 产品范围与发布门槛以 `PRD_v2.0.md` 为业务最高基线。
- 生产不确定性与降级策略以 `PRODUCTION_RUNTIME_ASSUMPTIONS_v1.0.md` 为运行基线。
- 工具库双层模型（部门技能与运行时集成）、界面缺口与待立项能力候选的**展开说明**以 `DESIGN_COVERAGE_AND_EXTENSIONS_v1.0.md` 为准；纳入 `AGENT_ABILITY_SPEC_v1.0.md` 前须完成正式能力 ID 与 YAML 条目。
- 全局 UI 顶栏（壳导航）约定以 `UI_UX_DESIGN_v1.0.md`「全局壳导航规范」为准。
- 营销中心「运营收入」与财务「账载收入」的口径区分与对账关系以 `DATA_ANALYTICS_INSIGHT_v1.0.md` 第 1.1 节为准。
- 培训中心课程素材的**全网发现、合规与入库**以 `TRAINING_CENTER_SPEC_v1.0.md` 为准（主脑主导检索，合法合规优先）。

## 三、文档更新要求

1. 更新设计文档时，必须同步检查以下基线文档：
   - `AGENT_ABILITY_SPEC_v1.0.md`
   - `DESIGN_COVERAGE_AND_EXTENSIONS_v1.0.md`（涉及技能库结构、运行时工具、UI 扩展或待立项能力时）
   - `quality_standards.md`
   - `PERFORMANCE_METRICS_v1.0.md`
   - `API_REFERENCE_v1.0.md`
2. 发现冲突时，在变更文档显式写明“以XX文档为基线”。
3. 禁止保留无效占位符（如示例仓库地址、无定义能力ID）。
4. 新增章节时，不重复整份文档一级标题，直接新增模块二级标题。
5. 超过 3000 行的单文档必须提供拆分计划，并在总览索引中登记。

## 四、发布前检查清单

- [ ] 术语一致（纪光元生、L0-L6）
- [ ] 能力ID可在能力规范中检索到
- [ ] API路径书写规则一致
- [ ] 验收与性能阈值一致
- [ ] 部署步骤可直接执行
- [ ] 无历史路径或无效占位符残留
- [ ] 超长文档已登记拆分计划与迁移状态
- [ ] 范围与门槛与 `PRD_v2.0.md` 一致
- [ ] 关键流程具备降级/回滚/人工接管说明
